## 核心规则

### 1. 代码质量规则

#### 1.1 类型安全
- 禁止使用类型压制（`as any`、`@ts-ignore`、`@ts-expect-error`）
- 优先使用显式类型而非隐式类型
- 在 TypeScript 项目中使用严格空值检查
- 在 TypeScript 配置中启用严格模式

#### 1.2 错误处理
- 禁止使用空 catch 块 `catch(e) {}`
- 始终记录带有适当上下文的错误
- 在 UI 组件中实现适当的错误边界
- 向用户返回有意义的错误消息

#### 1.3 测试要求
- 禁止删除失败的测试来"通过"
- 在实现功能之前先写测试（TDD 方法）
- 关键模块保持至少 80% 的代码覆盖率
- 为 API 端点包含集成测试

### 2. Git 工作流规则

#### 2.1 提交标准
- 使用约定式提交信息：`type(scope): description`
- 类型：feat、fix、docs、style、refactor、test、chore
- 保持提交原子性和专注性
- 在提交信息中引用 issue 编号

#### 2.2 分支管理
- 所有更改使用功能分支
- 遵循命名：`feature/描述`、`fix/描述`、`refactor/描述`
- 禁止直接提交到 main/master
- 合并前需要代码审查

#### 2.3 代码审查规则
- 合并前审查所有更改
- 检查安全漏洞
- 验证测试覆盖率
- 确保文档已更新

### 3. 项目结构规则

#### 3.1 文件组织
- 大型项目遵循领域驱动设计（DDD）
- 将相关文件放在一起（组件 + 样式 + 测试）
- 使用一致的命名规范
- 文件长度最多限制为 300 行

#### 3.2 配置管理
- 禁止将 secrets 提交到版本控制
- 对敏感数据使用环境变量
- 跨环境保持配置一致
- 记录所有配置选项

## 必备技能

### 1. 技术技能

#### 1.1 语言能力
- TypeScript/JavaScript（必备）
- Python（推荐）
- Go（推荐）
- Rust（可选）

#### 1.2 框架专长
- React/Next.js 用于前端
- Node.js/Express 用于后端
- Spring Boot 用于 Java 项目
- Django/FastAPI 用于 Python 项目

#### 1.3 数据库技能
- SQL（PostgreSQL、MySQL）
- NoSQL（MongoDB、Redis）
- ORM 使用（Prisma、TypeORM、SQLAlchemy）
- 数据库迁移管理

### 2. 开发工作流技能

#### 2.1 TDD 开发
- 先写失败的测试
- 实现最少代码通过测试
- 在保持测试覆盖率的同时重构
- 使用红-绿-重构循环

#### 2.2 代码审查
- 请求审查前先进行自我审查
- 提供建设性的反馈
- 带示例提出改进建议
- 遵循编码标准检查清单

#### 2.3 调试
- 使用适当的调试工具
- 有效阅读日志
- 创建最小复现案例
- 应用系统性调试方法

### 3. 工具技能

#### 3.1 IDE 和编辑器
- VS Code 配置和扩展
- Git 集成
- 终端效率
- 代码片段管理

#### 3.2 构建工具
- Webpack/Vite 用于前端
- Gradle/Maven 用于 Java
- Cargo 用于 Rust
- npm/yarn/pnpm 包管理

#### 3.3 测试工具
- Jest/Mocha 用于单元测试
- Cypress/Playwright 用于 E2E 测试
- Postman 用于 API 测试
- Lighthouse 用于性能测试

## 钩子系统

### 1. 预提交钩子

#### 1.1 代码检查
```
- 对暂存的文件运行 ESLint
- 检查 TypeScript 类型
- 验证代码格式（Prettier）
- 扫描敏感信息泄露
```

#### 1.2 测试
```
- 对更改的文件运行单元测试
- 检查测试覆盖率阈值
- 验证没有破坏性更改
- 验证集成测试
```

#### 1.3 质量门禁
```
- 检查提交信息格式
- 验证分支命名
- 确保 issue 引用
- 验证更新日志更新
```

### 2. 预推送钩子

#### 2.1 安全扫描
```
- 扫描漏洞（npm audit）
- 检查依赖许可证
- 验证敏感信息检测
- 扫描容器镜像
```

#### 2.2 构建验证
```
- 运行完整测试套件
- 构建生产工件
- 验证 Docker 镜像
- 检查包大小限制
```

### 3. CI/CD 钩子

#### 3.1 流水线阶段
```
1. 代码质量检查
   - Linting
   - 类型检查
   - 代码复杂度分析

2. 测试执行
   - 单元测试
   - 集成测试
   - E2E 测试（可选）

3. 安全扫描
   - 依赖漏洞
   - 静态分析
   - 敏感信息检测

4. 构建和部署
   - Docker 构建
   - 工件上传
   - 部署触发
```

#### 3.2 质量门禁
- 最低 80% 测试覆盖率
- 无严重/高危漏洞
- 所有测试通过
- 代码审查已批准

## 质量标准

### 1. 代码质量指标

#### 1.1 复杂度
- 圈复杂度 < 10
- 函数长度 < 30 行
- 类长度 < 200 行
- 文件长度 < 300 行

#### 1.2 可维护性
- 代码重复 < 5%
- 注释比例 10-20%
- 命名一致性得分 > 90%
- 模块耦合度得分 < 50%

#### 1.3 性能
- 首次内容绘制 < 1.5s
- 可交互时间 < 3s
- 包大小 < 500KB（初始）
- API 响应时间 < 200ms

### 2. 测试标准

#### 2.1 测试覆盖率
- 单元测试：最低 80%
- 集成测试：最低 60%
- 关键路径：100%

#### 2.2 测试质量
- 每个测试有单一断言焦点
- 测试独立且隔离
- Mock 外部依赖
- 使用描述性测试名称

#### 2.3 测试执行
- 单元测试：< 5 分钟
- 集成测试：< 15 分钟
- E2E 测试：< 30 分钟

## 最佳实践

### 1. 开发工作流

#### 1.1 开始任务
```
1. 彻底理解需求
2. 创建任务分解（todos）
3. 检查代码库中的现有模式
4. 设置开发环境
5. 先写测试（TDD）
```

#### 1.2 开发期间
```
1. 保持更改专注和原子性
2. 频繁运行测试
3. 定期提交
4. 更新文档
5. 尽早寻求反馈
```

#### 1.3 完成任务
```
1. 运行完整测试套件
2. 进行自我审查
3. 请求代码审查
4. 处理反馈
5. 合并和部署
```

### 2. 沟通标准

#### 2.1 提交信息
```
feat(auth): 添加 JWT 令牌刷新机制
- 实现令牌刷新端点
- 添加令牌过期处理
- 更新认证中间件
Fixes #123
```

#### 2.2 代码注释
```typescript
// 好：解释为什么，而不是做什么
// 使用闭包在异步回调中保留上下文
// 这种方法避免了事件处理程序中的绑定问题
function handleClick() {
  // ...
}

// 不好：冗余的解释
// 这个函数记录消息
function log(message) {
  console.log(message);
}
```

#### 2.3 文档
- API 端点：OpenAPI/Swagger 规范
- 数据库：实体关系图
- 架构：系统设计文档
- 运行手册：操作程序

### 3. 安全标准

#### 3.1 认证
- 使用行业标准认证（OAuth 2.0、JWT）
- 实现适当的会话管理
- 使用安全密码哈希（bcrypt、Argon2）
- 对敏感操作启用 MFA

#### 3.2 数据保护
- 静态加密敏感数据
- 传输中使用 TLS
- 实现适当的输入验证
- 清理输出以防止 XSS

#### 3.3 依赖安全
- 定期更新依赖
- 扫描漏洞（Snyk、Dependabot）
- 生产中使用固定版本
- 审查许可证合规性

## 技能实施指南

### 1. 设置技能

```bash
# 安装必备技能
npx skills add opencode/tdd-development
npx skills add opencode/code-review
npx skills add opencode/auto-reviewer

# 配置钩子
npm install -D husky lint-staged
```

### 2. 技能使用

```typescript
// 加载 TDD 开发技能
skill(name="tdd-development")

// 在实施期间
// 1. 先写失败的测试
// 2. 实现最少代码
// 3. 重构
// 4. 运行测试
// 5. 提交
```

### 3. 自定义技能

- 在 `skill.config.json` 中覆盖默认配置
- 添加项目特定规则
- 使用自定义脚本扩展钩子
- 与现有工具集成
