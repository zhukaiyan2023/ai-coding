# 代码审查技能

## 概述

此技能提供系统的代码审查方法，专注于代码质量、安全性、性能和最佳实践。它帮助审查者提供建设性的反馈，并帮助作者准备好代码供审查。

## 使用场景

- 审查拉取请求
- 提交更改前进行自我审查
- 执行同行评审
- 评估代码质量
- 检查安全漏洞

## 审查清单

### 1. 代码质量

- [ ] 代码遵循项目风格指南
- [ ] 变量和函数名称描述性强
- [ ] 函数小且专注（单一职责原则）
- [ ] 无代码重复（>5% 阈值）
- [ ] 注释解释为什么，而不是做什么
- [ ] 代码可读且自文档化

### 2. 类型安全

- [ ] 无 `any` 类型（无充分理由）
- [ ] 无 `@ts-ignore` 或 `@ts-expect-error`
- [ ] TypeScript 严格模式已启用
- [ ] API 有适当的类型定义
- [ ] 适当使用泛型

### 3. 错误处理

- [ ] 无空 catch 块
- [ ] 错误带有上下文记录
- [ ] 用户友好的错误消息
- [ ] 使用适当的异常类型
- [ ] 错误正确传播

### 4. 安全性

- [ ] 无硬编码的 secrets
- [ ] 实现了输入验证
- [ ] 防止 SQL 注入
- [ ] 避免 XSS 漏洞
- [ ] 检查了认证/授权
- [ ] 敏感数据已加密

### 5. 性能

- [ ] 无 N+1 查询问题
- [ ] 适当使用缓存
- [ ] 适当使用懒加载
- [ ] 优化了包大小
- [ ] 无内存泄漏

### 6. 测试

- [ ] 测试覆盖率符合要求
- [ ] 测试独立
- [ ] 覆盖了边界情况
- [ ] Mock 了外部依赖
- [ ] 测试可维护

### 7. 文档

- [ ] API 文档完整
- [ ] 需要时更新了 README
- [ ] 解释了复杂逻辑
- [ ] 记录了破坏性更改

## 审查类型

### 1. 快速审查（< 5 分钟）

对于小的更改、拼写错误、小修复：

```
✅ 检查：是否符合预期工作？
✅ 检查：无明显错误？
✅ 检查：代码风格一致？
```

### 2. 标准审查（15-30 分钟）

对于功能更改、错误修复：

```
✅ 完整清单审查
✅ 安全考虑
✅ 性能影响
✅ 测试覆盖率
```

### 3. 深度审查（1+ 小时）

对于架构更改、关键组件：

```
✅ 完整标准审查
✅ 设计模式分析
✅ 可扩展性考虑
✅ 集成测试
✅ 文档审查
```

## 提供反馈

### 好的评论

```typescript
// ✅ 好：具体且可操作
// 考虑使用 Map<string, User> 实现 O(1) 查询
// 而不是 array.find() 的 O(n)

// ✅ 好：解释原因
// 使用事件委托处理动态元素
// 这比给每个项目附加监听器更高效

// ✅ 好：问题礼貌
// 你认为将其提取到工具函数中怎么样？
// 它可以提高可重用性。
```

### 不好的评论

```typescript
// ❌ 不好：模糊
// 这很糟糕

// ❌ 不好：人身攻击
// 谁写的？太糟糕了。

// ❌ 不好：没有上下文
// 使用 XXX 代替
```

## 审查响应模板

### 对于作者

```
## 自我审查总结

- [ ] 我已在本地测试了更改
- [ ] 所有测试通过
- [ ] 我考虑了边界情况
- [ ] 我更新了文档

## 执行的测试

- 单元测试：✅
- 集成测试：✅
- 手动测试：✅（描述你测试的内容）

## 相关问题

- Fixes #123
- Related to #456
```

### 对于审查者

```
## 代码审查：[PR 标题]

### 总体印象
积极开始，然后是改进领域

### 必须修复（阻塞）
- [ ] 安全问题：第 45 行 - SQL 注入风险
- [ ] 错误：第 78 行 - 空指针异常

### 应该修复（推荐）
- [ ] 性能：考虑缓存此结果
- [ ] 可读性：提取为命名常量

### 挑剔（可选）
- [ ] 小问题：第 12 行多余空行
- [ ] 样式：可以使用 const 而不是 let

### 问题
- [ ] 你为什么选择这种方法？
- [ ] 这如何处理边界情况 X？

### 表扬
- 很好地使用 TypeScript 泛型！
- 关注点分离清晰。
- 测试覆盖率优秀！
```

## 反模式

### 对于审查者

- ❌ 不要挑剔格式（使用 linters）
- ❌ 不要阻止个人偏好
- ❌ 不要一次审查大量更改
- ❌ 不要对问题保持沉默
- ❌ 不要忘记表扬好的代码

### 对于作者

- ❌ 不要把反馈个人化
- ❌ 不要争论样式（使用 linters）
- ❌ 不要提交大型 PR
- ❌ 不要跳过自我审查
- ❌ 不要忽略阻塞性问题

## 最佳实践

### 提交前

1. 在本地运行所有测试
2. 运行 linters 和 formatters
3. 自我审查更改
4. 写清晰的 PR 描述
5. 保持 PR 原子性和专注性

### 审查期间

1. 在本地测试更改
2. 检查边界情况
3. 验证安全影响
4. 考虑性能影响
5. 提供建设性的反馈

### 审查后

1. 响应所有评论
2. 进行要求的更改
3. 重新请求审查
4. 回答问题
